# Uber NYC Operations Center — Assignment 5

## Quick Start

```bash
# 1. Clone the repo
git clone <repo-url>
cd HW5

# 2. Create and activate conda environment
conda create -n uber_nyc python=3.11 -y
conda activate uber_nyc
pip install pandas pyarrow tqdm requests

# 3. Download raw TLC trip data (~400 MB, one time)
python download_data.py

# 4. Preprocess into dashboard-ready JSON (~30 sec, one time)
python preprocess.py

# 5. Generate the DOW heatmap reference file (one time, ~2 sec)
python generate_heatmap.py

# 6. Launch the dashboard
python serve.py
# → opens http://localhost:8050 in your browser
```

> **JS libraries are bundled** — `static/libs/` already contains Plotly.js and Leaflet.js. Weather and events data (`static/data/weather.json`, `static/data/events.json`) are pre-generated and included in the repo — no API keys required.

## Overview

This dashboard simulates the real-time operations view that Uber's NYC General Manager would use to monitor ride performance, revenue health, geographic demand, and driver economics. It replays two years of actual TLC trip data (Jan 2023 – Dec 2024) as an animated time-series, advancing hour by hour with configurable playback speed.

---

## Assignment Choices & Rationale

### Data Choice: Contemporary TLC data (2023–2024) vs. 2014–2015

We chose the contemporary dataset because:
- It includes **pricing data** (`base_passenger_fare`, `driver_pay`) — critical for any revenue or economics view
- Two full calendar years (731 days) give strong seasonal signal: both winter slowdowns and summer/holiday peaks are visible
- The 2014–2015 data lacks fare information and covers only 9 months

The Uber subset is identified by `hvfhs_license_num = HV0003` in the High Volume For-Hire Vehicle Trip Records.

### Dashboard Elements (5 distinct elements, satisfying the 3–5 recommendation)

| # | Element | Question it Answers | GM Decision it Supports |
|---|---------|-------------------|------------------------|
| 1 | **24-HR Demand Pulse** — today vs. yesterday hourly chart | How is today tracking vs. the day before, hour by hour? | Intraday supply planning, anomaly detection |
| 2 | **Same-Weekday Forecast** — amber forecast + green actual | Is this hour running above or below what a typical [DOW] looks like? | Demand expectation-setting, driver positioning |
| 3 | **Borough Breakdown** — animated horizontal bars | Where is demand concentrated right now? | Geographic driver allocation, surge zone targeting |
| 4 | **NYC Live Demand Map** — animated bubble map + day/night tiles | Which boroughs are hottest, visually? | Visual hotspot identification, event impact |
| 5 | **90-Day Momentum** — daily rides + 7-day rolling avg | Is overall demand growing, flat, or declining? | Strategic trend identification, seasonality, growth tracking |

The **GM Alerts**, **Revenue Ops**, and **KPI row** are supporting elements that tie the five panels together into actionable signals.

### Why not Tableau/Streamlit?

We evaluated Streamlit but encountered fundamental limitations: full-page Python reruns on every slider move produced jerky animation and broken HTML rendering. Tableau does not natively support animated time playback. We built a pure HTML/CSS/JavaScript frontend because:
- `setInterval` + `requestAnimationFrame` enables **frame-rate-smooth** animation
- Pre-computed JSON files mean **zero server computation** during playback — latency is <1ms per frame
- The result is a completely self-contained static site that any browser can run

## System Architecture

```

        │
        ▼
data/uber_2023_agg.csv  (198 MB)
data/uber_2024_agg.csv  (200 MB)
data/taxi_zone_lookup.csv
        │
        ▼
preprocess.py           ← joins zone→borough, computes hourly stats, writes JSON
        │
        ▼
static/data/
  ├── meta.json              ← list of 731 dates, borough names, date range
  ├── daily.json             ← 731 rows: {date, rides, rides_wow, wow_pct}
  ├── weather.json           ← 731 days of NYC weather (pre-generated, included)
  ├── events.json            ← 20 major NYC events 2023–2024 (pre-generated, included)
  ├── dow_heatmap.json       ← 7×24 avg rides matrix (generated by generate_heatmap.py)
  └── by_date/
        └── YYYY-MM-DD.json  (× 731)   ← one file per calendar day
        │
        ▼
serve.py                ← Python http.server on localhost:8050
        │
        ▼
Browser: index.html + style.css + app.js
         Leaflet.js (map)  +  Plotly.js (charts)  — served from static/libs/
```

### Why pre-compute 731 JSON files?

Loading all data at once would require the browser to parse ~14 MB of JSON on startup. Instead, each date file is ~8–12 KB. The browser fetches today + yesterday + last-week + up to 4 same-weekday peer dates in parallel (`Promise.all`) — a handful of small requests totaling ~60–80 KB. Date navigation is instant; playback never blocks.

### Playback Engine

```
setInterval(tick, speedMs)          ← advances S.hour by 1 each tick
  └── if hour wraps past 23:        ← advance date index, fetch next date JSON
      await loadDateRange()         ← loads today + yest + lw + up to 4 peer dates
  └── updateAll()
        ├── updateHeader()          ← clock face, date, live/paused badge
        ├── updateKPIs()            ← 4 headline cards with animated number tween
        ├── updateWeatherBadge()    ← icon, hi/lo temp, precip for today
        ├── updateDemandChart()     ← Plotly.react() with 280ms transition
        ├── updateBoroBar()         ← CSS width transition (GPU-accelerated)
        ├── updatePeerChart()       ← same-weekday forecast vs. today
        ├── updateEco()             ← revenue ops + per-ride grids
        ├── updateAlerts()          ← rule-based GM signal engine
        ├── updateMap()             ← requestAnimationFrame ease-out marker resize
        ├── updateEventMarker()     ← gold event circle, time-gated, volume-scaled
        └── updateTrendCursor()     ← Plotly.react() rolling 90-day window
```

Speed options: ½× (2 s/hr) → 1× (1 s/hr) → 2× (500 ms/hr) → 4× (250 ms/hr) → 10× (100 ms/hr). At 10×, a full year plays back in ~24 seconds.

## Data Pipeline Details

### `download_data.py`
Downloads monthly Parquet files from NYC TLC CDN for 2023 and 2024. For each month:
- Filters rows where `hvfhs_license_num == "HV0003"` (Uber)
- Aggregates by `pickup_date × hour × PULocationID`
- Computes: `total_rides`, `avg_base_passenger_fare`, `avg_driver_pay`, `avg_trip_miles`
- Writes `data/uber_2023_agg.csv` and `data/uber_2024_agg.csv`

### `preprocess.py`
Joins aggregated CSVs with `taxi_zone_lookup.csv` (LocationID → Borough). For each of the 731 calendar dates:
- Groups by `hour × borough`
- Computes hourly borough-level rides, avg fare, avg driver pay, avg miles
- Writes `static/data/by_date/YYYY-MM-DD.json`

Also writes:
- `meta.json`: sorted list of all 731 dates + borough list
- `daily.json`: total daily rides + 7-day lag (rides_wow) for the trend chart

### `generate_heatmap.py`
Reads all 731 `by_date/` files and computes the average rides per (day-of-week × hour) cell. Output is `static/data/dow_heatmap.json` — a 7×24 reference matrix used as a fallback and for internal ranking. Run once after preprocessing.

### `static/data/weather.json` (pre-generated)
731 days of NYC weather fetched from the [Open-Meteo Historical API](https://open-meteo.com/) (free, no API key). Fields per date: `{cat, icon, hi, lo, precip, snow}`. Categories: clear, cloudy, fog, drizzle, rain, storm, snow.

### `static/data/events.json` (pre-generated)
20 major NYC events across 2023–2024 that materially affect Uber demand or road access. Fields per event: `{name, loc, borough, lat, lon, icon, note, start_h, end_h, demand_effect}`. The `demand_effect` field is `"elevates"` (NYE, Pride, etc.) or `"reduces"` (NYC Marathon road closures, Thanksgiving parade) and drives context-aware alert language.

## Complete KPI & Metric Reference

Every number visible on the dashboard is documented below with its exact calculation and the business reason it matters to a NYC GM.

---

### Header

#### Simulated Clock
- **Display**: `HH:MM AM/PM · DAY · MON DD, YYYY`
- **Calculation**: derived from `S.dates[S.idx]` (the current date in the dataset) and `S.hour` (0–23, the current playback hour). Not the real wall clock.
- **Business rationale**: Creates a first-person perspective — the GM sees the dashboard "as of" a specific moment, not as a historical summary. This makes temporal patterns visceral (watching demand drop off at 3 AM vs. explode at 5 PM).

#### LIVE / PAUSED Badge
- **LIVE** (red, pulsing): playback is running — hours are advancing automatically
- **PAUSED** (gray): playback is stopped — the GM is exploring a specific moment

---

### KPI Row — 4 Headline Cards

#### Card 1: RIDES THIS HOUR

| Field | Detail |
|-------|--------|
| **Main value** | `sum of rides across all boroughs for (current_date, current_hour)` |
| **Source** | `by_date/YYYY-MM-DD.json → hours[h].rides` |
| **▲/▼ vs yesterday** | `(today_rides − yesterday_rides) / yesterday_rides × 100%` where yesterday = same clock hour, date − 1 |
| **% of today's demand** | `today_hour_rides / sum(today_all_hours_rides) × 100%` |

**Business rationale**: The single most watched number for a GM. Volume is the top-line health signal. The delta vs. yesterday tells whether today is running hot or cold at this specific hour — a −20% reading at 6 PM on a Friday is an immediate red flag worth investigating (weather? service outage? competitor promotion?). The "% of today's demand" contextualizes it: 5% at 3 AM is normal; 5% at 6 PM is a crisis.

---

#### Card 2: AVG RIDER FARE

| Field | Detail |
|-------|--------|
| **Main value** | `avg_base_passenger_fare` for (current_date, current_hour), averaged across all boroughs weighted by ride count |
| **Source** | `by_date/YYYY-MM-DD.json → hours[h].avg_fare` |
| **▲/▼ vs yesterday** | `(today_fare − yesterday_fare) / yesterday_fare × 100%` |
| **X.X mi avg trip** | `avg_trip_miles` for this hour — see Per-Ride Economics |

**Note**: `base_passenger_fare` is the rider-facing price before tips, tolls, Black Car Fund surcharge, and the NYC Congestion Surcharge. It is not the total amount charged to the rider's card, but it is the controllable pricing signal.

**Business rationale**: Fare level determines both rider demand elasticity and gross revenue per trip. A fare rising alongside rising rides = healthy surge pricing working. A fare rising while rides fall = potential over-pricing or supply shortage causing rides to be declined. A fare falling while rides grow = pricing may be too low, leaving revenue on the table. The avg miles sub-line is essential context: a $30 fare on a 2-mile trip vs. a $30 fare on a 10-mile trip tell very different stories.

---

#### Card 3: DEMAND VS PEAK HOUR

| Field | Detail |
|-------|--------|
| **Main value** | `current_hour_rides / peak_hour_rides × 100%` |
| **Peak source** | `peakOf(S.lw ?? S.yest ?? S.today)` — busiest hour from last week's same day (falls back to yesterday, then today if last-week data unavailable) |
| **Sub-text** | Dynamically reflects actual data source: "last week", "yesterday", or "today's data" |

**Design decision**: We deliberately use last week's same day rather than today's actual peak, because a real GM cannot see the future. Last week's pattern is the best available predictor of today's intraday shape. The sub-label is dynamic — on the first 7 days of the dataset where last-week data doesn't exist yet, the label correctly says "yesterday" or "today's data" rather than claiming a comparison that isn't available.

**Business rationale**: Driver shift scheduling is the primary use case. Knowing that peak is 4 hours away at 6 PM lets the GM send driver incentive notifications at 4:30 PM to ensure supply is available before the spike. Seeing that current demand is already at 90% of last week's peak at 2 PM suggests today may be unusually strong — worth activating standby drivers early.

---

#### Card 4: SAME HOUR LAST WEEK

| Field | Detail |
|-------|--------|
| **Main value** | `(current_hour_rides − last_week_same_hour_rides) / last_week_same_hour_rides × 100%` |
| **Source** | `S.lw` (last week's date file, same hour index) |
| **Color** | Green if ≥ 0%, red if < 0% |

**Business rationale**: Day-over-day comparisons are noisy because Mondays look different from Fridays. Week-over-week at the same hour controls for day-of-week seasonality, making it the cleanest signal of true growth or decline. A GM seeing −10% WoW for three consecutive Fridays at 9 PM has a structural problem, not a weather blip. This metric is the dashboard's most important leading indicator of sustained trend changes.

### Left Panel

#### 24-HR Demand Pulse

| Element | Detail |
|---------|--------|
| **Green filled area (bright)** | Rides per hour for hours 0 → current_hour of today |
| **Green filled area (dim)** | Rides per hour for hours current_hour+1 → 23 of today (full day already in data) |
| **Blue-gray dotted line** | Yesterday's full 24-hour ride profile |
| **Vertical dashed line** | Current hour marker |
| **Green dot** | Exact position of current hour on today's curve |
| **Weather badge** | Today's conditions (icon, hi/lo temp, precipitation) |
| **Source** | `S.today.hours[*].rides` and `S.yest.hours[*].rides` |

**Business rationale**: The intraday shape matters as much as the total. A GM needs to see whether today's curve is tracking above or below yesterday's at every hour — not just the current snapshot. The split bright/dim fill makes elapsed vs. upcoming hours visually distinct. The weather badge provides immediate context for demand anomalies (rain → elevated demand; major snowstorm → constrained supply).

---

#### Same-Weekday Forecast

| Element | Detail |
|---------|--------|
| **Amber dashed line** | Forecast: per-hour average of the last N occurrences of this same day-of-week (e.g. last 4 Thursdays) |
| **Bold green line** | Today's actual hourly rides |
| **Vertical dashed line** | Current hour marker |
| **Readout below chart** | "[DOW] [hour] · Today: X,XXX rides · N-[DOW] avg: X,XXX · ±X.X% vs recent baseline" |
| **Source** | `S.peerData` — fetched in parallel with today's data on every date change |

**Data integrity**: The forecast is computed only from dates strictly before the current date — it never uses future data. On the first 1–3 weeks of the dataset where fewer than 4 prior same-weekday dates exist, the forecast uses however many are available and the title reflects the actual count (e.g. "LAST 2 THURSDAYS").

**Business rationale**: "Is demand normal for a Thursday?" is a fundamentally different question from "Is demand normal vs. yesterday?" This chart answers the former. The forecast line is the average of what the last N same-weekday hourly curves looked like — if today's green line consistently tracks above the amber line, it's a genuinely strong day for this day-of-week, not just a rebound from a weak yesterday.

---

#### Borough Breakdown · This Hour

| Element | Calculation |
|---------|------------|
| **Bar width** | `borough_rides / total_all_borough_rides × 100%` |
| **% label** | Same as bar width |
| **Ride count** | Absolute rides for that borough in the current hour |
| **Leader (bright green)** | Borough with highest absolute rides this hour |
| **Others (dim green)** | All other boroughs |
| **Source** | `hours[h].boroughs[borough].rides` |

**Business rationale**: Geographic demand concentration is the key input to driver deployment decisions. If Manhattan is at 38% of rides but only 25% of available drivers are positioned there, the GM should redirect. If Brooklyn's share suddenly spikes from 25% to 40%, an event (concert at Barclays, for example) is likely driving localized demand — warranting a geo-targeted driver incentive push in that borough.

---

#### Revenue Ops · This Hour

All four metrics are **hourly totals** — the cumulative economic impact of every ride completed in this one hour across all five boroughs.

| Metric | Formula | Source fields |
|--------|---------|---------------|
| **Gross Bookings** | `rides × avg_fare` | `hours[h].rides`, `hours[h].avg_fare` |
| **Uber Net Revenue** | `rides × (avg_fare − avg_pay)` | All three fields above |
| **Driver Earnings** | `rides × avg_pay` | `hours[h].rides`, `hours[h].avg_pay` |
| **Rides / Min** | `rides ÷ 60` | `hours[h].rides` |

**Gross Bookings**: The top-line GMV number. Tracks total rider payments processed this hour.

**Uber Net Revenue**: Platform contribution margin this hour, before corporate overhead. Catches margin erosion from unusual driver pay patterns.

**Driver Earnings**: Uber's largest single cost. Watching this relative to Gross Bookings reveals payout-ratio drift early.

**Rides / Min**: Operational tempo metric. Sudden drops during a normally busy period are an early signal of an app outage or data pipeline issue.

---

#### Per-Ride Economics · This Hour

| Metric | Formula | Source field |
|--------|---------|-------------|
| **Rider Base Fare** | `avg_base_passenger_fare` (weighted avg) | `hours[h].avg_fare` |
| **Driver Earnings/Ride** | `avg_driver_pay` (weighted avg) | `hours[h].avg_pay` |
| **Avg Miles** | `avg_trip_miles` (weighted avg) | `hours[h].avg_miles` |
| **Avg Trip Time** | `avg_trip_time / 60` minutes | `hours[h].avg_trip_time` |
| **Platform Margin** | `(avg_fare − avg_pay) / avg_fare × 100` | Derived |
| **Rev per Mile** | `avg_fare / avg_miles` | Derived |

**Platform Margin**: Uber's take rate — the percentage of rider fare the platform retains. Below 18% suggests margin pressure; above 28% may attract regulatory attention or driver dissatisfaction. The key metric a NYC GM would be held accountable for in quarterly reviews.

---

#### GM Alerts · This Hour

The alert engine fires rule-based signals every simulated hour. All signals include recommended actions.

| Rule | Threshold | Style | Trigger |
|------|-----------|-------|---------|
| Event upcoming | `S.hour < ev.start_h` | 🟡 Gold / ⏰ | Forewarns GM of imminent event impact |
| Event active | `start_h ≤ S.hour ≤ end_h` | 🟡 Gold / ev.icon | Confirms event is currently affecting demand |
| Demand vs yesterday ↑ | today > 1.15 × yesterday | 🟢 Hot / 📈 | Capacity pressure risk |
| Demand vs yesterday ↓ | today < 0.85 × yesterday | 🔵 Cold / 📉 | Demand stimulation opportunity |
| Week-over-week swing | \|WoW\| > 20% | 🟢/🟡 / 🚀⚠️ | Structural shift signal |
| Peak hour now | current hour = forecast peak | 🟢 Hot / ⚡ | Driver activation reminder |
| Margin low | margin < 18% | 🟡 Warning / 💸 | Unit economics alert |
| Margin high | margin > 30% | 🟢 Hot / ✅ | Efficiency signal |
| Forecast deviation ↑ | actual > forecast + 8% | 🟢 Hot / 📊 | Above-forecast demand (notes event if applicable) |
| Forecast deviation ↓ | actual < forecast − 8% | 🟡 Warning / 📊 | Below-forecast; if event `demand_effect = "reduces"`, alert says "Not a demand issue — road closures" |
| Rain / storm | weather cat in {rain, storm} | 🟢 Hot / 🌧️ | Demand spike expected |
| Snow | weather cat = snow | 🟡 Warning / ❄️ | Demand spike but supply constrained |
| Drizzle | weather cat = drizzle | 🟢 Hot / 🌦️ | Moderate demand uplift |
| Heat wave | hi ≥ 90°F | 🟢 Hot / 🌡️ | Heat-driven demand |
| Extreme cold | lo ≤ 20°F | 🟡 Warning / 🥶 | Cold-weather supply risk |
| All clear | no rules triggered | 🟢 OK / ✅ | Confirms normal operating envelope |

**Context-aware forecast alerts**: When demand is below forecast due to a known supply-constraining event (e.g. NYC Marathon road closures on 11/5), the alert reads "expected due to [event] road closures restricting driver routing — Not a demand issue" rather than the generic "consider demand stimulation." This avoids sending misleading operational guidance.

### Right Panel

#### NYC Live Demand Map

| Element | Calculation |
|---------|------------|
| **Inner circle radius** | `MIN_R + (borough_rides / max_borough_rides) × (MAX_R − MIN_R)` where MIN_R=6px, MAX_R=55px |
| **Outer glow radius** | `inner_radius × 1.6` |
| **Animation** | Ease-out cubic interpolation via `requestAnimationFrame` over 420ms |
| **Day/night tiles** | CartoDB Light (day) fades over CartoDB Dark (night) based on real sunrise/sunset for NYC on that date. Transition is ±0.5 hr around solar events with 450ms animation. |
| **Event marker** | Gold circle at event location, visible only during `[start_h, end_h]`. Radius scales with current borough ride volume relative to event-hour peak (R_MIN=10, R_MAX=42px). |
| **Source** | `hours[h].boroughs[borough].rides`; solar times computed from day-of-year + NYC latitude |

**Business rationale**: The map provides immediate spatial intuition that bar charts cannot. The day/night tile transition makes the time-of-day context visually obvious during playback. The gold event marker draws the GM's attention to the specific borough affected by a major event, calibrating their interpretation of the borough breakdown and demand alerts.

#### 90-Day Momentum

| Element | Calculation |
|---------|------------|
| **Window** | 90 calendar days ending on the current simulated date |
| **Green filled area** | `daily.json → rides` for each day in the window |
| **Blue dotted line** | 7-day rolling average — smooths day-of-week noise to reveal underlying trend |
| **Weather shading** | Blue tint for rain/storm days; lighter blue for snow days |
| **Green dot** | Today's position in the window |

**Design decision**: The prior version showed "same day last week" and "forecast" lines — but those comparisons are now fully covered by the Same-Weekday Forecast panel. This chart has a different purpose: macro momentum. The 7-day rolling average removes the weekly seasonality signal (which Fridays are always busier than Mondays), leaving only the underlying growth or decline trend.

**Business rationale**: The GM's strategic view. A rolling average that has been climbing for 6 weeks signals sustained market growth — worth presenting to leadership. A rolling average that has plateaued or reversed is the earliest warning of a structural demand problem, visible here weeks before it shows up in monthly reports. Weather shading explains anomalous single-day dips (a snow day with depressed rides is noise; a non-weather dip warrants investigation).

## Week-over-Week Implementation

Per the assignment suggestion, WoW comparisons are implemented in four places:

| Location | Calculation | Scope |
|----------|------------|-------|
| KPI Card 4 | `(hour_rides_today − hour_rides_last_week) / hour_rides_last_week × 100` | Hourly, same clock hour |
| KPI Card 1 delta | `(hour_rides_today − hour_rides_yesterday) / hour_rides_yesterday × 100` | Day-over-day (1-day lag) |
| 24-HR Demand Pulse | Overlay of yesterday's full 24-hour profile (blue-gray dotted) | Visual shape comparison |
| 90-Day Momentum | 7-day rolling average over daily totals | Macro trend signal |

## Forecasting

The same-weekday forecast method is used in two places:

| Location | Method | Scope |
|----------|--------|-------|
| Same-Weekday Forecast chart | Per-hour average of last N same-weekday dates | 24-hr hourly resolution |
| GM Alerts — forecast deviation | Daily ride total avg of last 4 same-weekday dates, scaled to elapsed hours | Daily total, intraday pacing |

**Why same-weekday instead of 7-day rolling average?** A rolling average mixes different days of the week (Mondays vs. Fridays), which have fundamentally different demand profiles. Using only prior occurrences of the same weekday controls for this seasonality and produces a more accurate baseline. The chart and alert thresholds only activate after hour 6 to avoid noisy signals when too little of the day has elapsed.

## Geographic Visualization

The assignment explicitly recommends geographic visualization using location data. We implement this via:
- **Borough-level animated bubbles** on a live Leaflet map using CartoDB Dark/Light tiles (no API key required)
- **Day/night tile switching** using a real solar position calculation (day-of-year, NYC latitude 40.71°N) — the map is visually light during daylight hours and dark at night, with a smooth 450ms crossfade centered on actual sunrise/sunset times
- **Event markers**: on dates with major NYC events, a gold circle appears at the event's geographic location. Its size scales with the current borough's ride volume relative to the event's peak hour — so it visually communicates both presence and intensity

A zone-level choropleth (263 individual TLC zones) was considered but rejected because borough-level aggregation is the natural granularity of GM decision-making.

## Data Limitations & Honest Caveats

- **No real-time data**: This is a historical replay, not a live feed. A production GM dashboard would ingest from Uber's internal data warehouse with ~5 min latency.
- **Borough-level only**: We aggregate to 5 boroughs. Zone-level (263 zones) detail exists in the raw data but was not visualized.
- **No competitor data**: The assignment suggests including taxi data as a competitor proxy. Yellow/green cab TLC data is available in the same portal; we did not include it to keep scope manageable.
- **Driver supply is unobserved**: `avg_driver_pay` and `avg_trip_miles` are proxies for supply conditions, but active driver count is not in the public TLC data.
- **Fare = base passenger fare**: This is the pre-tip, pre-fee amount. Total rider cost including tolls, Black Car Fund, and tips is higher.
- **Platform Margin is an approximation**: `(avg_fare − avg_pay) / avg_fare` approximates Uber's take rate but omits fixed costs, insurance, and regulatory fees that are also deducted from gross bookings before Uber books revenue.
- **Weather data is historical, not real-time**: `weather.json` was fetched once from Open-Meteo's historical API. In a live deployment this would be a real-time feed.
- **Events are manually curated**: The 20 events in `events.json` were selected based on known NYC calendar events that historically affect Uber demand. Smaller or unplanned events are not captured.
